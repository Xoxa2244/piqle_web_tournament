# Сценарий тестирования Piqle Partner API для IndyLeague

## Участники тестирования

1. **Партнер** — организация, интегрирующаяся с Piqle через API
2. **Турнирный директор** — пользователь Piqle, управляющий турниром через веб-интерфейс
3. **Супер админ** — администратор Piqle, управляющий партнерами и их доступом

---

## Этап 1: Подготовка и настройка

### Действия Супер Админа

1. **Создание партнерского аккаунта**
   - Войти в систему как супер админ
   - Перейти в раздел "Partner Integrations" (`/superadmin/partners`)
   - Создать нового партнера:
     - Название: "Test Partner Organization"
     - Email турнирного директора: `director@test-partner.com`
     - Среда: `SANDBOX`
     - Scope: `indyleague:write`
   - Сохранить и получить:
     - Partner Code: `test-partner-001`
     - API Key ID: `pk_...`
     - API Secret: `sk_...`

2. **Проверка настроек партнера**
   - Убедиться, что партнер связан с турнирным директором
   - Проверить, что API ключи активны
   - Убедиться, что scope `indyleague:write` назначен

### Действия Турнирного Директора

1. **Регистрация в системе**
   - Создать аккаунт на платформе Piqle (если еще не создан)
   - Email: `director@test-partner.com`
   - Выполнить вход в систему
   - Убедиться, что доступ к панели управления турнирами есть

### Действия Партнера

1. **Получение учетных данных**
   - Получить от супер админа:
     - Partner Code
     - API Key ID
     - API Secret
     - Base URL: `https://rtest.piqle.io/api/v1/partners/indyleague`

2. **Настройка тестового окружения**
   - Подготовить Postman коллекцию или другой инструмент для тестирования
   - Настроить переменные окружения:
     - `base_url`: `https://rtest.piqle.io/api/v1/partners/indyleague`
     - `api_key_id`: `pk_...`
     - `api_secret`: `sk_...`
     - `external_tournament_id`: `test-tournament-{{$timestamp}}`
   - Настроить заголовки для всех запросов:
     - `Authorization: Bearer {api_key_id}:{api_secret}`
     - `Content-Type: application/json`
     - `Idempotency-Key: {uuid}` (для POST запросов)

## Этап 2: Создание турнира

### Действия Партнера

1. **Создание турнира**
   - Отправить POST запрос на `/tournaments/upsert`
   - Тело запроса:
     ```json
     {
       "externalTournamentId": "test-tournament-2024-01-15",
       "name": "Test IndyLeague Tournament",
       "seasonLabel": "Winter 2024",
       "timezone": "America/New_York"
     }
     ```
   - **Ожидаемый результат:**
     - Статус: `200 OK`
     - Ответ содержит `internalTournamentId`
     - Статус: `"created"`

2. **Проверка создания турнира**
   - Повторить тот же запрос с тем же `externalTournamentId`
   - **Ожидаемый результат:**
     - Статус: `200 OK`
     - Статус: `"updated"` (не `"created"`)

### Действия Турнирного Директора

1. **Проверка появления турнира**
   - Войти в систему Piqle
   - Перейти в раздел "My Tournaments"
   - Убедиться, что турнир "Test IndyLeague Tournament" появился в списке
   - Открыть турнир и проверить:
     - Название: "Test IndyLeague Tournament"
     - Формат: IndyLeague
     - Season Label: "Winter 2024"

### Действия Супер Админа

1. **Проверка в админке**
   - Перейти в раздел партнеров
   - Открыть партнера "Test Partner Organization"
   - Проверить, что турнир создан и связан с правильным турнирным директором

## Этап 3: Создание дивизионов

### Действия Партнера

1. **Создание дивизионов**
   - Отправить POST запрос на `/divisions/upsert`
   - Тело запроса:
     ```json
     {
       "externalTournamentId": "test-tournament-2024-01-15",
       "divisions": [
         {
           "externalDivisionId": "div-mens-a",
           "name": "Men's A",
           "orderIndex": 1
         },
         {
           "externalDivisionId": "div-womens-a",
           "name": "Women's A",
           "orderIndex": 2
         }
       ]
     }
     ```
   - **Ожидаемый результат:**
     - Статус: `200 OK`
     - Ответ содержит массив `items` с двумя дивизионами
     - Оба со статусом `"created"`

### Действия Турнирного Директора

1. **Проверка дивизионов**
   - Открыть турнир в веб-интерфейсе
   - Перейти в раздел "Divisions"
   - Убедиться, что видны два дивизиона:
     - "Men's A"
     - "Women's A"
   - Проверить порядок отображения (должен соответствовать `orderIndex`)

### Действия Партнера

1. **Создание команд**
   - Отправить POST запрос на `/teams/upsert`
   - Тело запроса:
     ```json
     {
       "externalTournamentId": "test-tournament-2024-01-15",
       "teams": [
         {
           "externalTeamId": "team-alpha",
           "divisionExternalId": "div-mens-a",
           "name": "Team Alpha",
           "clubName": "Alpha Club",
           "eventType": "men"
         },
         {
           "externalTeamId": "team-beta",
           "divisionExternalId": "div-mens-a",
           "name": "Team Beta",
           "clubName": "Beta Club",
           "eventType": "men"
         },
         {
           "externalTeamId": "team-gamma",
           "divisionExternalId": "div-mens-a",
           "name": "Team Gamma",
           "clubName": "Gamma Club",
           "eventType": "men"
         },
         {
           "externalTeamId": "team-delta",
           "divisionExternalId": "div-mens-a",
           "name": "Team Delta",
           "clubName": "Delta Club",
           "eventType": "men"
         }
       ]
     }
     ```
   - **Ожидаемый результат:**
     - Статус: `200 OK`
     - Ответ содержит массив `items` с четырьмя командами
     - Все со статусом `"created"`

### Действия Турнирного Директора

1. **Проверка команд**
   - Открыть дивизион "Men's A"
   - Убедиться, что видны 4 команды:
     - Team Alpha
     - Team Beta
     - Team Gamma
     - Team Delta
   - Проверить, что у каждой команды отображается название клуба

## Этап 5: Создание игроков

### Действия Партнера

1. **Создание игроков (32 игрока для 4 команд по 8 человек)**
   - Отправить POST запрос на `/players/upsert`
   - Тело запроса содержит 32 игрока (по 8 на каждую команду)
   - Пример для одной команды:
     ```json
     {
       "externalTournamentId": "test-tournament-2024-01-15",
       "players": [
         {
           "externalPlayerId": "player-alpha-001",
           "firstName": "John",
           "lastName": "Doe",
           "email": "john.doe@example.com",
           "gender": "M",
           "duprId": "12345",
           "phone": "+1234567890",
           "externalTeamId": "team-alpha"
         },
         {
           "externalPlayerId": "player-alpha-002",
           "firstName": "Jane",
           "lastName": "Smith",
           "email": "jane.smith@example.com",
           "gender": "M",
           "externalTeamId": "team-alpha"
         }
         // ... еще 6 игроков для team-alpha
         // ... 8 игроков для team-beta
         // ... 8 игроков для team-gamma
         // ... 8 игроков для team-delta
       ]
     }
     ```
   - **Ожидаемый результат:**
     - Статус: `200 OK`
     - Ответ содержит массив `items` с 32 игроками
     - Все со статусом `"created"`

2. **Проверка ограничения на количество игроков**
   - Попытаться добавить 9-го игрока в команду (должно быть ограничение до 8)
   - **Ожидаемый результат:**
     - Статус: `422` или предупреждение в ответе
     - Сообщение об ошибке о превышении лимита

### Действия Турнирного Директора

1. **Проверка игроков**
   - Открыть команду "Team Alpha"
   - Убедиться, что видны все 8 игроков
   - Проверить отображение:
     - Имя и фамилия
     - Email (если указан)
     - DUPR рейтинг (если указан)
   - Повторить для остальных команд

     ```json
           "statusHint": "scheduled"
         }

     }
     ```
   - **Ожидаемый результат:**
     - Статус: `200 OK`
     - Ответ содержит массив `items` с двумя днями
     - Оба со статусом `"created"`

### Действия Турнирного Директора

1. **Проверка дней матчей**
   - Открыть турнир
   - Перейти в раздел "Match Days"

   - **Ожидаемый результат:**

     - Сообщение о необходимости 4 активных игроков

1. **Проверка ростеров**

     - Все 8 игроков команды
     - 4 активных игрока (отмечены как активные)
     - Буквы (A/B/C/D) еще не назначены (будут назначены вручную)

## Этап 9: Назначение букв и ввод результатов (Турнирный директор)

### Действия Турнирного Директора

1. **Назначение букв игрокам**
   - Открыть матч "Team Alpha vs Team Beta"
   - В разделе "Roster Management":
     - Для Team Alpha: назначить буквы A, B, C, D четырем активным игрокам
     - Для Team Beta: назначить буквы A, B, C, D четырем активным игрокам
   - Нажать "Save Roster"
   - **Ожидаемый результат:**
     - Ростер сохранен
     - Буквы отображаются рядом с именами игроков
     - Статус матча изменился на `READY` или `IN_PROGRESS`

2. **Генерация игр**
   - На странице "Score Input" (`/admin/{tournamentId}/stages`)
   - Выбрать дивизион "Men's A"
   - Выбрать день "15 февраля 2024"
   - Нажать кнопку "Generate Games" для дивизиона
   - **Ожидаемый результат:**
     - Создано 12 игр для каждого матча
     - Игры отображаются в интерфейсе
     - Статус матчей изменился на `IN_PROGRESS`

3. **Ввод результатов игр**
   - Открыть вкладку с матчем "Team Alpha vs Team Beta"
   - Для каждой из 12 игр ввести счет:
     - Например: Game 1: 11-9 (Home), Game 2: 9-11 (Away), и т.д.
   - **Ожидаемый результат:**
     - Счет сохраняется сразу (оптимистичное обновление)
     - Нет задержки при вводе
     - Иконки победы/поражения отображаются корректно

4. **Обработка ничьей (6-6)**
   - Ввести результаты так, чтобы общий счет был 6-6
   - **Ожидаемый результат:**
     - Появляется поле для выбора победителя tie-break
     - Можно выбрать Home или Away команду
     - После выбора матч помечается как завершенный

5. **Завершение матча**
   - После ввода всех результатов и (при необходимости) tie-break
   - **Ожидаемый результат:**
     - Статус матча: `COMPLETED`
     - Общий счет отображается корректно
     - Статистика обновлена

## Этап 10: Получение результатов через API

### Действия Партнера

1. **Получение статуса дня**
   - Отправить GET запрос на `/days/day-001`
   - **Ожидаемый результат:**
     - Статус: `200 OK`
     - Ответ содержит:
       - `totalMatchups`: 2
       - `completedMatchups`: 2 (если оба матча завершены)
       - `matchupsRequiringTieBreak`: 0 или 1 (если был tie-break)
       - `matchupsWithMissingScores`: 0
       - `isReady`: true

2. **Получение результатов дня**
   - Отправить GET запрос на `/days/day-001/results`
   - Опционально: добавить параметр `?divisionExternalId=div-mens-a`
   - **Ожидаемый результат:**
     - Статус: `200 OK`
     - Ответ содержит:
       - `divisionResults` с массивом результатов по дивизионам
       - Для каждой команды:
         - `wins`, `losses`
         - `pointsFor`, `pointsAgainst`, `pointDiff`
       - Детали матчей (если запрошены)

3. **Проверка агрегации**
   - Убедиться, что статистика команд корректна:
     - Количество побед/поражений
     - Сумма очков (PF/PA)
     - Разница очков (DIFF)

## Этап 11: Обновление данных через API

### Действия Партнера

1. **Обновление информации о команде**
   - Отправить POST запрос на `/teams/upsert` с обновленными данными:
     ```json
     {
       "externalTournamentId": "test-tournament-2024-01-15",
       "teams": [
         {
           "externalTeamId": "team-alpha",
           "divisionExternalId": "div-mens-a",
           "name": "Team Alpha Updated",
           "clubName": "Alpha Club Updated",
           "eventType": "men"
         }
       ]
     }
     ```
   - **Ожидаемый результат:**
     - Статус: `200 OK`
     - Статус: `"updated"` (не `"created"`)

2. **Добавление нового игрока в команду**
   - Отправить POST запрос на `/players/upsert` с новым игроком:
     ```json
     {
       "externalTournamentId": "test-tournament-2024-01-15",
       "players": [
         {
           "externalPlayerId": "player-alpha-009",
           "firstName": "New",
           "lastName": "Player",
           "email": "new.player@example.com",
           "gender": "M",
           "externalTeamId": "team-alpha"
         }
       ]
     }
     ```
   - **Ожидаемый результат:**
     - Статус: `422` или предупреждение (команда уже полная - 8 игроков)
     - Или игрок добавлен, если команда была не полной

### Действия Турнирного Директора

1. **Проверка обновлений**
   - Обновить страницу турнира
   - Убедиться, что изменения от партнера отобразились:
     - Название команды обновлено
     - Новый игрок появился (если был добавлен)

## Этап 12: Тестирование ограничений и ошибок

### Действия Партнера

1. **Попытка обновить завершенный матч**
   - Попытаться обновить матч со статусом `COMPLETED`
   - **Ожидаемый результат:**
     - Статус: `422` или `400`
     - Сообщение об ошибке: "Cannot update completed matchup"

2. **Попытка создать ростер с несуществующим игроком**
   - Отправить POST запрос на `/rosters/upsert` с `externalPlayerId`, которого нет
   - **Ожидаемый результат:**
     - Статус: `422`
     - Сообщение об ошибке: "Player not found"

3. **Попытка использовать неверный API ключ**
   - Отправить запрос с неверным `Authorization` header
   - **Ожидаемый результат:**
     - Статус: `401`
     - Сообщение об ошибке: `INVALID_API_KEY`

4. **Попытка использовать неверный Idempotency-Key**
   - Отправить два одинаковых запроса с разными `Idempotency-Key`
   - **Ожидаемый результат:**
     - Оба запроса обработаны (разные ключи = разные запросы)

5. **Попытка использовать один Idempotency-Key дважды**
   - Отправить два одинаковых запроса с одним `Idempotency-Key`
   - **Ожидаемый результат:**
     - Первый запрос обработан
     - Второй запрос возвращает тот же результат (идемпотентность)

---

## Этап 13: Очистка тестовых данных

### Действия Партнера

1. **Удаление турнира через cleanup endpoint**
   - Отправить DELETE запрос на `/tournaments/test-tournament-2024-01-15/cleanup`
   - **Ожидаемый результат:**
     - Статус: `200 OK`
     - Ответ содержит информацию об удаленных сущностях:
       - tournament: 1
       - divisions: 2
       - teams: 4
       - players: 32
       - matchDays: 2
       - matchups: 2
       - externalMappings: "all related"

## Дополнительные сценарии тестирования

### Сценарий A: Регенерация игр

1. **Турнирный директор:**
   - Генерирует игры для матча
   - Вводит несколько результатов
   - Нажимает "Regenerate Games"
   - **Ожидаемый результат:**
     - Появляется предупреждение о сбросе результатов
     - После подтверждения:
       - Все игры удалены и созданы заново
       - Все результаты сброшены
       - Tie-break winner сброшен
       - Статус матча: `IN_PROGRESS`

### Сценарий B: Удаление матча

1. **Турнирный директор:**
   - Открывает день матчей
   - Нажимает "Delete Matchup" для матча со статусом `PENDING`
   - Подтверждает удаление
   - **Ожидаемый результат:**
     - Матч удален
     - Ростеры для этого матча также удалены

2. **Попытка удалить завершенный матч:**
   - **Ожидаемый результат:**
     - Кнопка "Delete Matchup" неактивна или отсутствует
     - Или ошибка при попытке удаления

### Сценарий C: Обновление ростеров после назначения букв

1. **Турнирный директор:**
   - Назначает буквы игрокам вручную
   - Сохраняет ростер

2. **Партнер:**
   - Обновляет ростер через API (меняет активных игроков)
   - **Ожидаемый результат:**
     - Ростер обновлен
     - Буквы для новых активных игроков назначены автоматически (A/B/C/D по порядку)
     - Буквы для игроков, которые остались активными, сохранены

## Критерии успешного тестирования

✅ Все эндпоинты API работают корректно  
✅ Идемпотентность соблюдается  
✅ Валидация данных работает правильно  
✅ Ошибки обрабатываются с понятными сообщениями

✅ Партнер может получать результаты после ввода их директором  
✅ Обновления данных через API корректно отображаются в веб-интерфейсе  
✅ Cleanup endpoint удаляет все связанные данные  
✅ Ограничения (8 игроков на команду, 4 активных в ростере) соблюдаются

## Примечания

- Все тесты должны выполняться в среде `SANDBOX`
- После каждого теста рекомендуется очищать данные через cleanup endpoint
- При тестировании в `PRODUCTION` необходимо быть осторожным с cleanup операциями
- Рекомендуется использовать Postman коллекции для автоматизации тестирования
- Все внешние ID должны быть уникальными в рамках партнерского аккаунта
