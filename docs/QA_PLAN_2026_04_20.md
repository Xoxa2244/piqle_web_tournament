### QA-1. Email OTP flow + rate limit

**Что проверяем:** security-фикс ship-blocker #2 — лимит 10 запросов / 10 мин per IP на `/request-code` и `/signup`. Плюс базовый OTP flow не сломан.

**Шаги:**
1. Разлогинься, открой `stest.piqle.io/auth/signin`
2. Введи свой email → получи код → введи → успешно войди ✓
3. Разлогинься. Попробуй запросить код **11 раз подряд** с одного IP (разные email'ы) в течение минуты
4. На 11-й запрос должен прийти HTTP 429 с сообщением "Too many code requests"
5. Попробуй ввести email вида `test@demo.iqsport.ai` или `foo@example.com` — должно отклонять

**Ожидаемый результат:**
- OTP flow работает end-to-end
- 11-й запрос/мин возвращает HTTP 429 с headers `Retry-After` + `X-RateLimit-Remaining: 0`
- Blocked domains не получают письма, возвращается ошибка

**Критерий бага:** если можно отправить >10 кодов/10 мин или если blocked domain получает реальное письмо — это regression.

### QA-2. Launch Runbook — полный цикл Go Live → Kill Switch

**Что проверяем:** Новая страница `/intelligence/launch` — главная точка контроля перед включением AI. Сегодня клубы первый раз ей пользуются.

**URL:** `stest.piqle.io/clubs/<test-club-id>/intelligence/launch`

**Шаги:**
1. Открой страницу — должна загрузиться без ошибок. Видишь секции: Automatic checks / Voice & tone / Preview / Manual confirmations / Go Live button
2. В **Automatic checks** должен быть список из ~11 пунктов с иконками (зелёная ✓ / жёлтая ⚠ / красная ✗). Внизу summary: "X passed · Y warnings · Z blockers"
3. Кнопка **Re-run checks** — нажми → крутилка 1-2 сек, список обновляется
4. В **Voice & tone**: поменяй tone на "Energetic", length на "Short", нажми **Save voice settings** → должен появиться зелёный "Saved ✓"
5. В **Preview**: кликни "Slot Filler" → LLM генерит example (5-10 сек loader) → появляется subject + body
6. Кликни **Too formal** → регенерирует → новый subject+body (должен звучать casual-нее)
7. Кликни **Shorter** → текст должен стать короче
8. Поставь все 6 **Manual confirmations** галочки
9. Кнопка **Go Live** должна стать активной (зелёная); без галок — disabled + пояснение "X confirmations remaining"
10. Кликни **Go Live** → (если нет red blockers) должен появиться badge "LIVE" вверху, появится секция **Kill Switch** (красная)
11. В **Audit log** должна появиться строчка "Went live — [твой username] — [timestamp]"
12. Кликни **Stop all AI sending** → появляется поле reason (минимум 3 символа) → введи "testing rollback" → **Confirm kill** → badge LIVE исчезает, в audit log — новая строчка "Kill switch"

**Ожидаемый результат:** весь цикл отрабатывает без ошибок. Go Live заблокирован когда есть errors или непоставленные галочки. Kill Switch всегда работает и требует reason (≥3 символа).

**Частные проверки:**
- Если есть red blockers в Automatic checks — Go Live серый и disabled даже с галочками
- Нажатие на actionLabel ("Set budget", "Verify DNS", "Run dry-run") ведёт на соответствующую страницу
- `?demo=true` в URL — страница должна работать, audit log пуст — норм

### QA-3. SQL Injection regression — Cohort filters

**Что проверяем:** ship-blocker #1 fix — `buildCohortWhereClause` должна безопасно отклонять malicious input.

**URL:** `stest.piqle.io/clubs/<id>/intelligence/cohorts`

**Шаги:**
1. Создай новый cohort. Добавь фильтр "Age" gte и значение `55` (нормальное число). Сохрани → показывает count matching members ✓
2. Попробуй значение `abc` (не число) → должен показать 0 members или ошибку
3. Попробуй значение `1' OR '1'='1` → **НЕ должен** показать всех членов клуба (это тест на SQL injection)
4. Попробуй значение `-5` или `10000` → должен показать 0
5. Для "Frequency" (sessions/month) → те же тесты: `abc`, `1'; DROP TABLE --`, `-5`
6. Для "Days Since Last Visit" → те же тесты

**Ожидаемый результат:**
- Валидные числа работают
- Невалидные / hostile → возвращают 0 members (не всю БД, не ошибку 500)
- В DevTools → Network tab ответ 200, не 500

**Критерий бага (P0 security):** если ввод `1' OR '1'='1` возвращает **всех** членов клуба — это дыра безопасности.

### QA-5. Email Domain setup → verify → enable flow

**URL:** `/clubs/<id>/intelligence/email-domain`

**Часть А — flow с dummy доменом:**
1. Зайди на чистом клубе (без настроенного домена). Должна быть форма "Add your domain"
2. Введи `test.example.com` → **Generate DNS records** → появляются 3 записи: SPF (TXT), DKIM (TXT), Return-Path (CNAME)
3. Каждая запись имеет **Copy** кнопку — нажми → в clipboard должно скопироваться значение (проверь paste в notepad)
4. Появляется статус-badge "Awaiting DNS" (оранжевый)
5. **Verify DNS** — нажми → через 3-5 сек ответ. Для `test.example.com` без реальных DNS → SPF "not detected", DKIM "not detected" → жёлтый статус
6. **Enable sending** должен быть скрыт/disabled пока не verified
7. **Remove domain** → confirm → поля очищаются, возвращается исходная форма

**Часть Б — невалидный ввод:**
8. Введи IP `192.168.1.1` → должно отклонить "IP addresses are not supported"
9. Введи `example.com` (root domain) → должно показать предупреждение "looks like a root domain. We recommend a subdomain"
10. Введи `not a domain` → отклонить "Not a valid domain format"

**Часть В — реальный домен (если доступен):**
Если у тебя есть доступ к тестовому домену с управлением DNS (Cloudflare/GoDaddy):
11. Настрой SPF + DKIM TXT записи → подожди 10-30 мин propagation
12. Нажми Verify → статус должен стать "Verified"
13. Нажми **Enable sending** → badge меняется на "Live"
14. Отправь тестовое письмо через Campaigns → в headers должно быть `From: campaigns@<твой-домен>`

**Ожидаемый результат:** flow работает, DNS records корректные, verified + enabled → реально шлёт с кастомного домена.

**Критерий бага:** если Enable доступен до Verify, или если кастомный From не применяется при отправке.

### QA-10. Cost tracking per-tenant

